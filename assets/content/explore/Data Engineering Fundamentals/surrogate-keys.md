# Surrogate Keys

A **surrogate key** is an artificial/internal key generated inside the data warehouse.

A **natural key** is a real business key coming from the source system.

Example:

|Key type|Example|Meaning|
|---|---|---|
|**Natural key**|`customer_id = CUST1001`|ID from CRM/source system|
|**Surrogate key**|`customer_key = 98765`|ID generated in the warehouse|

---

## Natural key vs surrogate key

| Point           | Natural Key                                           | Surrogate Key                                          |
| --------------- | ----------------------------------------------------- | ------------------------------------------------------ |
| Source          | Comes from source/business system                     | Generated by warehouse/data platform                   |
| Meaning         | Has business meaning                                  | No business meaning                                    |
| Example         | `customer_id`, `national_id`, `email`, `product_code` | `customer_key`, `product_key`, auto-increment ID, UUID |
| Stability       | Can change                                            | Should never change                                    |
| Uniqueness      | May not be globally unique                            | Designed to be unique                                  |
| Used for joins? | Sometimes, but not ideal in DW                        | Preferred for fact-to-dimension joins                  |
| Good for        | Identifying business entity                           | Linking facts to correct dimension version             |

---

## Why surrogate keys are used in dimensions

| Reason                          | Explanation                                                                   |
| ------------------------------- | ----------------------------------------------------------------------------- |
| **Handle SCD Type 2**           | Same customer can have multiple historical rows, each needs a different key   |
| **Avoid source system changes** | If source `customer_id` changes, warehouse joins stay stable                  |
| **Support multiple sources**    | Two systems may both have `customer_id = 1001` but mean different customers   |
| **Improve joins**               | Integer surrogate keys are usually smaller and faster than long business keys |
| **Hide business meaning**       | Internal key avoids exposing sensitive or unstable business identifiers       |
| **Manage unknown records**      | You can create special keys like `-1 = Unknown`, `-2 = Not Applicable`        |
| **Ensure consistency**          | Facts join to a controlled warehouse key instead of raw source keys           |


---

## Example: SCD Type 2 with surrogate key

Customer changes city from **Cairo** to **Dubai**.

|customer_key|customer_id|name|city|effective_date|expiry_date|current_flag|
|--:|---|---|---|---|---|---|
|101|CUST01|Ahmed|Cairo|2023-01-01|2024-06-30|N|
|205|CUST01|Ahmed|Dubai|2024-07-01|9999-12-31|Y|

Here:

|Column|Type|
|---|---|
|`customer_id`|Natural key|
|`customer_key`|Surrogate key|

The same natural key `CUST01` appears twice because the customer has two historical versions.

Facts should join to the correct `customer_key`.

Example:

|order_id|order_date|customer_key|amount|
|---|---|--:|--:|
|O1001|2024-05-10|101|500|
|O1002|2024-08-15|205|700|

This allows historical reporting:

- May order belongs to customer when city was **Cairo**
    
- August order belongs to customer when city was **Dubai**

## Key generation in distributed systems

In distributed systems, key generation is harder because multiple workers/nodes may insert data at the same time.

### Common approaches

| Method                           | Example                                 | Pros                                | Cons                                 |
| -------------------------------- | --------------------------------------- | ----------------------------------- | ------------------------------------ |
| **Database sequence / identity** | `IDENTITY`, `SEQUENCE`                  | Simple, ordered                     | Can become bottleneck                |
| **UUID**                         | `550e8400-e29b...`                      | Globally unique, no central service | Large, slower joins, not ordered     |
| **Snowflake-style ID**           | Timestamp + node id + sequence          | Unique, roughly ordered, scalable   | Needs clock coordination             |
| **Hash key**                     | Hash of natural key or business columns | Deterministic, easy to regenerate   | Collision risk, hard with SCD Type 2 |
| **Composite key**                | source_system + natural_key             | Useful for staging                  | Not ideal as DW surrogate key        |
| **Central key service**          | Dedicated service generates IDs         | Controlled and consistent           | Extra dependency                     |

---

## Distributed key generation comparison

| Approach               | Best used when                             | Interview comment                                             |
| ---------------------- | ------------------------------------------ | ------------------------------------------------------------- |
| **Sequence/identity**  | Single warehouse/database controls inserts | Good for traditional DW dimensions                            |
| **UUID**               | Many systems generate keys independently   | Good uniqueness, but heavy for joins                          |
| **Snowflake-style ID** | High-scale distributed ingestion           | Good balance of uniqueness and ordering                       |
| **Hash key**           | Data Vault or deterministic pipelines      | Useful, but must manage collisions and SCD logic              |
| **Composite key**      | Landing/staging layers                     | Useful for uniqueness, but not preferred for analytical joins |

## Important interview points

### 1. Natural key can change

Example: customer email changes.

Bad design:

```sql
fact_order.customer_email = dim_customer.email
```

Better design:

```sql
fact_order.customer_key = dim_customer.customer_key
```

Because email is a business attribute, not a stable warehouse key.

---

### 2. Same natural key can exist in multiple systems

|source_system|customer_id|customer_name|
|---|---|---|
|CRM|1001|Ahmed|
|ERP|1001|Different Customer|

So you often need:

```text
source_system + natural_key
```

to identify the business record before assigning a surrogate key.

---

### 3. Surrogate keys are critical for SCD Type 2

Without surrogate keys, you cannot easily distinguish multiple historical versions of the same customer.

```text
customer_id = CUST01
```

could refer to Cairo version or Dubai version.

But:

```text
customer_key = 101 → Cairo version
customer_key = 205 → Dubai version
```

is clear.

---

## Interview answer template

> A natural key comes from the business or source system, like customer_id or product_code. A surrogate key is generated inside the warehouse and has no business meaning. In dimensional modeling, surrogate keys are preferred because they are stable, support SCD Type 2, handle multiple source systems, improve joins, and allow special records like unknown or not applicable. In distributed systems, we can generate surrogate keys using sequences, UUIDs, Snowflake-style IDs, hashes, or a central key service, depending on scale and consistency requirements.


## Best practice

In dimensional modeling, **surrogate keys are usually generated for dimensions, not facts**.

Facts normally **store the dimension surrogate keys as foreign keys**.

Example:

|Table|Key behavior|
|---|---|
|`dim_customer`|Generates `customer_key`|
|`dim_product`|Generates `product_key`|
|`fact_sales`|Stores `customer_key`, `product_key`, `date_key`|

---

## Dimension surrogate keys

### Best ways to generate keys in dimensions

|Method|Best for|Pros|Cons|
|---|---|---|---|
|**Sequence / identity column**|Traditional data warehouse|Simple, fast, small integer keys|Harder in distributed systems|
|**Warehouse-generated ID**|Snowflake, Redshift, SQL Server, Oracle|Easy to manage centrally|Depends on warehouse engine|
|**Snowflake-style ID**|Distributed systems|Scalable, roughly ordered|More complex|
|**UUID**|Multi-system distributed pipelines|Globally unique|Large key, slower joins|
|**Hash key**|Data Vault, deterministic ELT|Repeatable, no lookup needed|Collision risk, not always ideal for SCD2|

---

## Recommended for dimensions

### 1. Classic warehouse / single database

Use **integer sequence / identity**.

```sql
customer_key BIGINT IDENTITY
```

Best because:

- small key
    
- fast joins
    
- easy indexing
    
- simple SCD Type 2 handling
    

Example:

|customer_key|customer_id|city|effective_date|current_flag|
|--:|---|---|---|---|
|1|C001|Cairo|2024-01-01|N|
|2|C001|Dubai|2025-01-01|Y|

Same natural key, different surrogate keys.

---

### 2. Distributed lakehouse / Spark / Databricks

Avoid relying on simple auto-increment from many workers.

Better options:

|Option|Use when|
|---|---|
|**Warehouse sequence generated during dimension merge**|You centralize dimension loading|
|**Identity columns in Delta/Snowflake/BigQuery where supported**|Platform supports safe generated keys|
|**Snowflake-style IDs**|Many writers need to generate keys independently|
|**Deterministic hash keys**|You need repeatable keys across environments|

For SCD Type 2, a useful hash pattern is:

```text
surrogate_key = hash(source_system, natural_key, effective_date)
```

or:

```text
surrogate_key = hash(source_system, natural_key, version_start_date)
```

This makes each historical version unique.

## Fact table keys

Facts usually do **not need their own surrogate key** unless required.

A fact row is often identified by:

```text
order_id + order_line_id
```

or by a combination of dimension keys:

```text
date_key + customer_key + product_key + store_key
```

### Best practice for facts

|Scenario|Recommended approach|
|---|---|
|Transaction fact|Use source transaction ID as degenerate dimension, plus dimension foreign keys|
|Periodic snapshot fact|Use grain columns like date + account + product|
|Accumulating snapshot fact|Use business process ID like order_id, claim_id, shipment_id|
|Need technical uniqueness|Add `fact_row_id` using sequence/identity/UUID|
|Need deduplication|Use natural business event key or hash of event fields|

---

## Fact surrogate key vs degenerate dimension

For facts, many interviewers expect this distinction:

|Concept|Meaning|Example|
|---|---|---|
|**Fact surrogate key**|Artificial technical row ID|`sales_fact_key = 987654`|
|**Degenerate dimension**|Business identifier stored in fact table|`order_id = O1001`, `invoice_id = INV55`|

In many dimensional models, `order_id` is stored directly in the fact table as a **degenerate dimension**, not as a separate dimension table.

---

## Recommended pattern

### Dimension table

```sql
dim_customer (
    customer_key BIGINT,        -- surrogate key
    source_system STRING,
    customer_id STRING,         -- natural key
    customer_name STRING,
    city STRING,
    effective_date DATE,
    expiry_date DATE,
    current_flag BOOLEAN
)
```

### Fact table

```sql
fact_sales (
    date_key INT,
    customer_key BIGINT,
    product_key BIGINT,
    store_key BIGINT,
    order_id STRING,            -- degenerate dimension
    quantity INT,
    sales_amount DECIMAL
)
```

The fact table stores `customer_key`, not `customer_id`.

---

## Loading facts correctly

When loading facts, you usually do a lookup from natural key to dimension surrogate key.

Example:

```sql
SELECT
    f.order_id,
    d.customer_key,
    f.product_key,
    f.order_date,
    f.amount
FROM staging_sales f
JOIN dim_customer d
  ON f.customer_id = d.customer_id
 AND f.order_date >= d.effective_date
 AND f.order_date < d.expiry_date;
```

This is very important for **SCD Type 2**.

The fact must join to the dimension version that was valid at the time of the event.

---

## Handling unknown dimension keys

Always design special surrogate keys.

|Key|Meaning|
|--:|---|
|`-1`|Unknown|
|`-2`|Not applicable|
|`-3`|Error / invalid|
|`-4`|Late arriving dimension|

Example:

If a sale arrives but customer is missing:

```text
customer_key = -1
```

Later, when the customer dimension arrives, you can update/reprocess the affected facts.

---

## Best recommendation by environment

|Environment|Best surrogate key strategy|
|---|---|
|Traditional EDW|Integer sequence / identity for dimensions|
|Snowflake / Redshift / SQL Server|Sequence or identity columns|
|BigQuery|Generate keys during controlled dimension build, often hash or sequence-like logic|
|Spark / Databricks|Use Delta identity if suitable, or deterministic hash keys|
|Data Vault|Hash keys are common|
|High-scale distributed ingestion|Snowflake-style IDs or UUIDs|
|Facts|Usually no surrogate key; use dimension FKs + degenerate business IDs|

---

## Interview answer

> The best practice is to generate surrogate keys in dimension tables and store those keys as foreign keys in fact tables. For dimensions, integer sequence or identity keys are usually best in a centralized warehouse because they are small and fast for joins. In distributed systems, UUIDs, Snowflake-style IDs, or deterministic hash keys can be used, but we need to consider ordering, collision risk, and join performance. Fact tables usually do not need their own surrogate key unless there is a technical requirement; instead, they should use dimension surrogate keys plus business transaction IDs as degenerate dimensions.
